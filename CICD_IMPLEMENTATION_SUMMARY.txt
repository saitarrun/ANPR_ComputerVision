GITHUB SECRETS REQUIRED
================================================================================

Set in Settings → Secrets and variables → Actions:

Staging Deployment:
  STAGING_DEPLOY_HOST     = staging-server.example.com
  STAGING_DEPLOY_USER     = deploy
  STAGING_DEPLOY_KEY      = (SSH private key, PEM format)

Production Deployment:
  PROD_DEPLOY_HOST        = prod-server.example.com
  PROD_DEPLOY_USER        = deploy
  PROD_DEPLOY_KEY         = (SSH private key, PEM format)

Notifications:
  SLACK_WEBHOOK_DEPLOY    = https://hooks.slack.com/services/...

Note: REGISTRY, IMAGE_NAME, and GITHUB_TOKEN are auto-configured.

ENVIRONMENT SETUP ON SERVERS
================================================================================

Each deployment server requires:

1. Ubuntu 20.04+ OS
2. Docker + Docker Compose installed
3. Nginx for load balancer (staging/prod)
4. SSH user "deploy" with key-based auth
5. PostgreSQL client tools
6. Secrets stored in .env.staging or .env.prod:
   - POSTGRES_PASSWORD (32-char random)
   - REDIS_PASSWORD (32-char random)
   - FERNET_KEY (from cryptography library)
   - JWT_SECRET (32-char random)
   - SECRET_KEY (32-char random)

See ops/DEPLOYMENT_SERVER_SETUP.md for complete walkthrough.

TYPICAL WORKFLOWS
================================================================================

DEVELOP A FEATURE
  1. git checkout -b feat/endpoint
  2. Code + test locally
  3. git push → Opens PR
  4. GitHub Actions: lint, test, security, build
  5. Get code review approval
  6. Merge to main
  7. GitHub Actions: auto-deploy to staging
  8. Verify on staging, manual testing

RELEASE TO PRODUCTION
  1. Ensure everything working on staging for 1+ hours
  2. git tag -a v0.1.0 -m "Release notes"
  3. git push origin v0.1.0
  4. GitHub Actions: deploy-prod.yml triggers
  5. Check email for manual approval request
  6. Approve deployment
  7. Blue-green deployment executes automatically:
     - Deploy to green on port 8001
     - Health check green
     - Switch LB to green
     - Monitor for 2 min
     - If healthy: keep green, shut down blue
     - If error: auto-rollback to blue
  8. Verify production: curl https://api.anpr.com/healthz

EMERGENCY ROLLBACK
  1. gh workflow run rollback-prod.yml \
       --ref main \
       --raw-field target_version=v0.0.5 \
       --raw-field reason="Error rate spike"
  2. Or use GitHub UI: Actions → Rollback Production → Run workflow
  3. Workflow executes:
     - Stop current deployment
     - Deploy target version
     - Health check
     - Switch LB back
     - Slack notification
  4. Verify: curl https://api.anpr.com/healthz
  5. See ops/RUNBOOK_ROLLBACK.md for detailed steps
