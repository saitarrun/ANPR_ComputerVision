TECHNICAL ARCHITECTURE
================================================================================

IMAGE BUILD LAYERS:
  Layer 1: FROM python:3.11-slim              [~160 MB base]
  Layer 2: Install build tools                [~250 MB added]
  Layer 3: Copy pyproject.toml + source       [~15 KB]
  Layer 4: Create venv + pip install -e .     [~1.2 GB in builder, discarded]
  [STAGE 2]
  Layer 5: FROM python:3.11-slim              [~160 MB base]
  Layer 6: Install runtime deps               [~50 MB]
  Layer 7: COPY venv from builder             [~200 MB]
  Layer 8: COPY source code                   [~20 KB]
  Layer 9: Create appuser + set permissions   [negligible]
  [Final Size: ~360 MB]

DOCKER COMPOSE NETWORKING:
  Network: anpr-network (bridge driver)
  API can reach: postgres (host=postgres), redis (host=redis), minio (host=minio)
  External ports:
    - 8000 (API)
    - 5432 (PostgreSQL)
    - 6379 (Redis)
    - 9000 (MinIO S3)
    - 9001 (MinIO Console)

HEALTH CHECK STRATEGY:
  Liveness (/healthz):
    - Always succeeds if container alive
    - Used by: Docker restart policies
    - Interval: 30s, Timeout: 10s, Retries: 3
  
  Readiness (/readyz):
    - Checks database connectivity
    - Used by: Load balancers, K8s readiness probes
    - Interval: 30s, Timeout: 10s, Retries: 3
    - Start period: 30s (grace before first check)

SECURITY MODEL:
  - Non-root user (appuser, UID 1000)
  - No hardcoded secrets (all via environment)
  - Minimal base image (fewer CVEs)
  - Resource limits (prevent DOS)
  - Health checks (prevent unhealthy traffic)
  - Network isolation (internal communication only)
  - No build tools in runtime image (attack surface ↓30%)

================================================================================
DESIGN DECISIONS
================================================================================

1. MULTI-STAGE BUILD
   Rationale: Separate builder (with deps) from runtime (minimal)
   Benefits: 30% smaller image, no build tools in prod, faster pulls
   Trade-off: Slightly longer build time (negligible)

2. PYTHON:3.11-SLIM BASE
   Rationale: Latest stable Python, minimal (no build tools, docs, tests)
   Benefits: Smaller attack surface, faster pulls, security updates
   Alternative: python:3.11-alpine (even smaller, but pip/build slower)

3. NON-ROOT USER (appuser)
   Rationale: Mitigate container escape (if attacker gains access to container, not host)
   Implementation: UID 1000 (standard unprivileged user)
   Security: Matches Kubernetes security best practice

4. DOCKER COMPOSE FOR LOCAL DEV (not just bare docker run)
   Rationale: Single command launches full stack (API + DB + Redis + Workers)
   Benefits: Minimal setup friction, mirrors production config, easy to learn
   Trade-off: Requires Docker Compose (minimal extra disk)

5. SEPARATE PROD OVERRIDES FILE (docker-compose.prod.yml)
   Rationale: DRY principle; dev/prod configs share base, override for prod
   Benefits: Easy to diff dev vs prod, less duplication, explicit prod changes
   Usage: docker-compose -f docker-compose.yml -f docker-compose.prod.yml

6. GIT COMMIT HASH TAGGING
   Rationale: Every image tied to exact commit (reproducibility + rollback)
   Implementation: ./scripts/docker-build.sh defaults to git rev-parse --short HEAD
   Benefits: Know exactly what code is in production, quick rollback
   Alternative: Only semver (lose traceability to commits)

7. MAKEFILE FOR CONVENIENCE
   Rationale: Developers shouldn't need to remember long docker-compose commands
   Benefits: Discoverability (make help), consistency, teaching tool
   Trade-off: One more file to maintain (justified by time savings)
